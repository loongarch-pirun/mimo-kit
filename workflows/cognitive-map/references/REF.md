## T0：工具基线 — 完整说明

### 定位

> LLM 写"这里看起来有问题"，工具写"这里确实有问题"。
> T0 是 LLM 分析与专业工具之间的显式交接层。

### 执行条件（逐项检测）

```
a. 竞态检测：
   检测到 Makefile/CMakeLists/go.mod/Cargo.toml
   → 检查编译选项/链接痕迹中是否含 -fsanitize=thread/-race
   → 若含，尝试编译运行测试套件并捕获 TSan/Go race 原始输出
   → 若不含但 Makefile 存在，检查是否可注入 -fsanitize=thread 到 CFLAGS
   → 若项目为 Go 且有 go.mod：go test -race ./...（Go race detector 内建）

b. 依赖审计：
   检测到 package-lock.json → 验证 npm audit 可用 → npm audit --json
   检测到 Cargo.lock → 验证 cargo audit 可用 → cargo audit --json
   检测到 requirements.txt/Pipfile → 验证 pip-audit 可用
   检测到 go.mod → 验证 govulncheck 可用
   检测到 pom.xml → 验证 mvn dependency-check:check 可用

c. IaC 安全扫描：
   检测到 .tf 文件 → 验证 tfsec/checkov 可用
   检测到 K8s YAML（deployment/service/pod 特征）→ 验证 kubescape/kube-linter 可用

d. 区块链：
   检测到 .sol 文件 → 验证 Slither/Mythril 可用
   检测到 Hardhat/Foundry 配置 → 验证对应框架测试套件

e. 若对应工具未安装或编译失败 → 标记【T0 未执行：$原因】，不阻塞后续
   T0 非阻塞——跳过仅意味着后续维度会产生更多【需 * 工具复核】标记

f. 时间层（非阻塞）：
   检测到 .git 目录 → 执行：
     git log --oneline --since="6 months ago" -- <项目根>
     git log --oneline --diff-filter=D --since="12 months ago" -- <项目根>
     git log --oneline -- <项目根> | wc -l  （总提交数）
   → 生成变更热力描述（≤100 tokens）：
     - 最近活跃区（3 个月内频繁变更的目录/文件）
     - 死区（archive/、注释排除的文件、>12 月无提交的目录及其最后活跃时间）
     - 近期趋势（增长/稳定/清理中）
   → 产出存档于 00-tool-baseline.md 的时间层摘要段
   → 注入到各维度扫描上下文（影响"模式"发现）
   → 若 git 不可用则跳过，标记【时间层跳过：git 不可用】
```

### T0 产出结构

```
00-tool-baseline.md
  ├── 原始输出（完整，存档用，不进入后续 prompt）
  └── LLM 可读摘要（≤200 tokens，仅含）：
        - 工具名、版本
        - 扫描范围（测试套件覆盖的函数/文件集，若能获取）
        - 高危项数量、关键坐标列表（仅文件:行号，不展开）
        - 明确标注【覆盖盲区】：工具未触及的模块/文件/函数
          （如"TSan 仅覆盖 *_test.go 调用的函数，以下模块无测试入口：pkg/unzip/, internal/parser/raw/"）

阶段B2 各维度仅接收"LLM 可读摘要"作为上下文。
禁止引用原始输出全文——原始输出仅存档于 00-tool-baseline.md。
```

### 为什么要摘要层

npm audit JSON 可能 500+ 行，TSan 报告可能 2000+ 行。直接塞进阶段B2 的 prompt：
- 撑爆上下文窗口
- LLM 可能把工具结论当成自己的分析，违反约束第 12 条
- 后续维度的发现质量被稀释（上下文被工具输出占据）

摘要层隔离原始输出，LLM 在阶段B2 只看到坐标列表和覆盖范围。

## 阶段B2：每维产出格式

每个发现的**强制输出格式**：

```
- 坐标：文件:行号:函数
- 描述：...
- 来源：【静态推断】或【工具已确认: 工具名 版本】
- 如为【静态推断】且涉及运行时行为 → 追加【需 * 工具复核】
- 就绪度：【就绪】/【需复现】/【供参考】
```

完整模板：

```markdown
# 维度{编号}：{名称}

## 发现
- 坐标：文件:行号:函数
  描述：...
  来源：【静态推断】
  标记：【需竞态检测工具复核】
  就绪度：【就绪】/【需复现】/【供参考】

## 盲区
- 无法确定的内容

## 需人工验证
- 【文件:行号】描述 + 验证建议

## 本维度发现分级

### ■ 锚定
坐标精确、代码可见、无歧义的结构事实。可直接引用。
- 【发现】

### ◆ 模式
跨文件/跨模块重复出现的特征，理解全局需要知道。
- 【发现】

### ○ 盲区
当前手段无法确认，需其他视角/工具/权限才能覆盖。
- 【发现】
```

## Agent 路由表

认知地图发现可被路由到的 agent（参考 `~/.claude/rules/common/agents.md`）：

| 发现类型 | Agent | 典型来源 |
|---------|-----------|---------|
| 安全漏洞（注入点、认证缺失、硬编码密钥） | `security-reviewer` | 维度7 锚定 |
| 结构问题（循环依赖、复杂度热点、跨层违规） | `architect` | 维度1 锚定 |
| 代码质量（反模式、命名偏差、重复代码） | `code-reviewer` | 维度8 模式 |
| 编译错误 | `build-error-resolver` | T0 确认 |
| 性能热点（CPU/内存/IO） | `performance-optimizer` | 维度6 锚定 |
| 测试盲区（源文件无对应测试） | `tdd-guide` | 维度11 锚定 |
| 死代码/冗余（未使用函数、重复依赖） | `refactor-cleaner` | 维度9/10 模式 |
| 运行时竞态 | 先复现，再交 `code-reviewer` | 维度5 盲区 |

路由规则：
- `【就绪】` → 直接路由到对应 agent
- `【需复现】` → 标记路由目标但不执行，在工具验证清单中列出复现条件
- `【供参考】` → 不路由，记录到技术债追踪

## LLM 与 T0 消歧规则（完整版）

阶段B2 各维度遇到与 T0 相关的发现时，按以下决策树处理：

```
1. 该代码路径是否被 T0 工具扫描覆盖？
   （对照 T0 摘要中明确声明的【覆盖盲区】判断，不猜测）
   → 若【已覆盖】且 T0 无发现：标记【T0 已覆盖：静态推断补充】
   → 若【未覆盖】或 T0 未执行：标记【需 * 工具复核】

2. LLM 静态发现与 T0 发现冲突时：
   → 以 T0 为准，LLM 发现降级为【○ 盲区】
   → 标注【与 T0 输出冲突，需人工仲裁】
   → 记录到工具验证清单的"冲突项"行

3. T0 发现、LLM 未注意到的路径：
   → 直接引用 T0 摘要，标注【T0 发现，LLM 确认】
   → 不重新分析——T0 工具已给出结论

4. T0 摘要层信息不足以判断覆盖范围时：
   → 保守处理，标记【T0 覆盖范围不明确，需人工确认】
   → 不得伪装已确认
```

## 十一维度完整问题清单

### 维度1：结构
- 核心调用链（A→B→C）及入口点
- 循环依赖/递归风险
- 复杂度热点（嵌套深、异常处理缺失）
- [时间层] 孤儿文件/死代码的最后活跃时间（来自 T0 git log）

### 维度2：契约
- 显性契约（签名/类型/文档）
- 隐性契约（调用约定）
- 契约漂移（文档 vs 实现不一致）

### 维度3：语义
- 业务意图（一句话）
- 主流程/异常/降级/补偿路径
- 状态机（若存在）及业务规则映射
- [编译器] AST 转换正确性、语言规范边界 Case、错误恢复路径、语法二义性处理
- [区块链] 交易原子性、Gas 消耗路径、fallback/receive 逻辑、权限升级路径

### 维度4：数据
- 关键数据生命周期
- 敏感数据位置（密码/Token/PII）
- 数据一致性风险（非事务操作、副本不一致）
- [ML] 特征泄露（训练/验证/测试分离）、数据分布漂移、标签质量、数据管道可复现性
- [区块链] 存储布局与状态变量碰撞（代理升级模式）、事件发射完整性、链上/链下数据一致性
- [移动端] 离线队列与冲突合并、本地 DB 迁移路径、Keychain/Keystore 凭据持久化

### 维度5：时序

**LLM 能做的（静态代码模式识别）：**
- 执行模型显式标注（sync/async/goroutine/promise/await 模式）
- 锁/原子操作/通道使用模式（是否每次共享访问都受保护）
- 超时/重试/退避策略缺失（HTTP 请求、数据库连接、RPC 调用）
- 回调地狱深度（>3 层嵌套回调/Then 链）

**LLM 做不了的（需运行时工具验证）：**
- 细粒度竞态窗口（data race 的确切位置）
- 实际死锁路径（运行时锁序图才能确定的循环等待）
- 内存序违规（C++ memory_order、无锁数据结构的正确性）
- TOCTOU（文件系统/共享内存的检查-使用时间窗口）

> 属于"LLM 做不了"的发现 → 标记【需竞态检测工具复核】
> T0 已执行竞态检测 → 对照 T0 摘要的覆盖盲区走消歧决策树

### 维度6：资源
- CPU/内存/IO 热点与瓶颈（通用）
- 外部配额（API 限流等）
- [游戏] GPU 显存/帧预算/Shader 编译开销、纹理/网格加载、物理引擎 CPU 占用
- [ML] GPU 显存/训练吞吐/推理延迟、数据加载管线瓶颈、混合精度训练稳定性
- [移动端] 电池消耗/后台进程限制/内存压力回调/网络类型切换(WiFi↔蜂窝)
- [桌面应用] 多窗口/多渲染器内存、原生互操作开销、长时运行进程内存泄漏

### 维度7：安全

**LLM 能做的（静态代码模式识别）：**
- 硬编码密钥/Token/密码/API Key（正则匹配）
- 注入点代码模式（字符串拼接构造 SQL/命令/LDAP/OS 命令）
- 不安全反序列化入口（pickle/JSON.parse 无 schema/unmarshal 无类型限制）
- CORS/重定向/SSRF 白名单缺失（URL 参数直传 fetch/http.get）

**LLM 做不了的（需专用安全扫描工具）：**
- 依赖库已知 CVE（需 CVE 数据库对照）
- 配置漂移（运行时配置与声明不一致）
- 运行时攻击面（实际可达的端点和参数）
- 加密算法强度（AES-ECB vs AES-GCM 等需密码学知识）

> 属于"LLM 做不了"的发现 → 标记【需专用安全扫描复核】
> 攻击面分析按领域自适应表选取检查路径（见下方）

#### 安全领域自适应检查路径

先识别项目攻击面形态，再选对应路径。**不要硬套 Web 后端四步法。**

| 领域 | 攻击面形态 | LLM 检查路径 | 工具优先 |
|------|-----------|-------------|---------|
| Web后端 | HTTP端点/WebSocket/文件上传 | 注入点(SQL/命令/模板)→密钥泄露→不安全反序列化 | semgrep/CodeQL |
| 前端 | DOM/用户输入/第三方脚本/API响应 | XSS→CSRF→依赖供应链→敏感数据泄露(localStorage) | eslint-plugin-security |
| 移动端 | 平台权限/跨App URL/推送/网络 | 权限模型过度授权→Keychain/Keystore安全→ATS/证书绑定 | Xcode Analyze / Android Lint |
| 嵌入式 | 外设输入/网络帧/调试接口/JTAG | 硬件攻击面(UART/JTAG/SPI)→固件签名验证→内存破坏 | cppcheck/coverity |
| 桌面应用 | IPC通道/原生互操作/自动更新/本地文件 | 沙箱逃逸→nodeIntegration→更新签名验证→本地文件访问越权 | 平台沙箱检测 |
| IaC | 资源定义/状态文件/模块源/TF变量 | IAM策略过度宽松→资源公开暴露→密钥明文(状态文件)→Provider供应链 | tfsec/checkov |
| 区块链 | 外部调用/交易排序/Oracle数据/初始化 | 重入→整数溢出→访问控制绕过→抢先交易/Oracle操纵 | Slither/Mythril |
| 编译器/语言工具 | 输入源码/编译选项/插件/宏展开 | 恶意输入触发UB→宏展开代码注入→插件沙箱逃逸→输出代码注入 | ASan/cargo-audit |
| 通用 | — | 攻击面→注入点→密钥泄露→依赖库CVE（默认四步法） | 按语言选取 |

### 维度8：规范
- 语言/框架惯用法偏差
- 反模式（上帝对象/魔法数字/重复代码）
- 误导性命名

### 维度9：变更
- 技术债标记（TODO/FIXME/临时绕过）
- 不兼容变更（接口对比）
- 回滚可行性（不可逆迁移等）
- [时间层] 对照 git log 的近期活跃区/死区，标注技术债标记的年龄（是否超过 6 个月未处理）

### 维度10：依赖

**LLM 能做的（静态代码模式识别）：**
- 依赖图拓扑（谁依赖谁，层级关系）
- 循环依赖检测（A→B→C→A）
- 版本号语义（lock 文件存在时可读版本范围）
- manifest 与 lock 不一致（package.json 声明 ^2.0 但 lock 锁定 1.8）
- 废弃包检测（npm deprecation notice / PyPI "development status: dead"）
- 未声明依赖（代码中使用但未在配置中声明的库）

**LLM 做不了的（需依赖审计工具）：**
- 已知 CVE（需 CVE 数据库对照）
- 许可证冲突（需许可证兼容性规则库）
- 供应链投毒/私有源替换（需包完整性校验）
- 传递依赖版本冲突（需完整依赖树解析）

> 属于"LLM 做不了"的发现 → 标记【需依赖审计工具复核】

**领域扩展：**
- [多服务/全栈] 跨服务共享依赖版本对齐、Docker 镜像基础层一致性、服务间耦合扇入/扇出
- [IaC] Provider SDK 版本锁定、社区模块注册源信任、Terraform 模块依赖树
- [桌面应用] 原生库版本绑定（OS SDK 最低版本）、自动更新依赖链

### 维度11：测试（仅记录可观察事实，不做质量判断）

可记录：
- ✅ 关键路径（入口/核心逻辑/异常处理）是否有对应的测试文件
- ✅ 测试文件 ↔ 源文件的映射关系（哪些源文件无对应测试）
- ✅ 阻碍可测试性的结构因素（单例/全局状态/硬编码依赖/private 方法不可达）
- ✅ 测试覆盖率盲区（基于文件映射，不含行级覆盖率）

不可记录（属于质量判断，违反约束）：
- ❌ "断言不够严格"
- ❌ "Mock 过度导致假通过"
- ❌ "只测了 happy path"
- ❌ "测试质量低"

## 增量模式

用户说"增量扫描,对比 A..B"时触发。

### 行为差异

1. **阶段A**：仅输出「变更摘要」——变更文件数量、目录分布、新增/修改/删除行数统计。不回答完整的6个问题
2. **T0**：仅重新执行受变更文件影响的工具子项（如仅变了 .sol 文件就只重跑 Slither，不变更 npm 项目就不重跑 npm audit）
3. **阶段B1**：仅更新受影响的高风险面，不重写全景
4. **阶段B2**：仅扫描变更文件涉及的维度项。有变更→正常输出；无变更→标注"本维度无变更影响"
5. **输出**：「认知地图 diff」——标注新增/消除/需重验

### 诚实局限

> 增量模式的调用方识别基于**静态可见引用**（grep 函数名/导入路径/类名）。以下调用关系**可能遗漏**：
> - 反射/动态派发（Python getattr、JS 动态 import、Java 反射）
> - 宏展开后的调用（C/C++ 函数式宏）
> - 字符串拼接的函数名/路由
> - 装饰器/中间件/拦截器等非显式调用链
>
> 高安全要求项目或变更涉及上述模式的，建议使用完整扫描。

### 增量产物模板

```markdown
# 增量认知地图 diff

## 变更范围
- 变更文件数：N
- 新增行数：+X
- 删除行数：-Y
- 静态可见受影响调用方：M（可能遗漏反射/动态派发/宏展开后的调用）

## 🆕 新增发现
- 【文件:行号:函数】发现

## ✅ 已消除
- 【文件:行号:函数】原发现（因代码变更而不再适用）

## ⚠️ 需重新验证
- 【文件:行号:函数】原发现（变更可能改变了判断前提）

## 遗漏声明
[如果项目中存在反射/动态派发/宏等不可静态追踪的调用模式，在此标注]
```

### 增量重扫验证闭环（Agent 集成）

修复完成后，增量重扫形成验证闭环：

```
1. git commit → 记录修复了哪些发现的坐标
2. 增量重扫变更文件（"增量扫描,对比 HEAD~1..HEAD"）
3. 对照原发现列表：
   - ✅ 已消除：原发现坐标不再匹配 → 从 交接单中勾除
   - ⚠️ 需重新验证：修复改变了上下文但未消除根因 → 更新发现描述
   - 🆕 新增：修复引入了新发现 → 进入 路由
4. 更新 交接单状态
```

此闭环确保每次修复都有可追溯的验证记录。

```
   → 若【未覆盖】或 T0 未执行：标记【需 * 工具复核】

2. LLM 静态发现与 T0 发现冲突时：
   → 以 T0 为准，LLM 发现降级为【供参考】
   → 标注【与 T0 输出冲突，需人工仲裁】

3. T0 发现、LLM 未注意到的路径：
   → 直接引用 T0 摘要，标注【T0 发现，LLM 确认】
```
