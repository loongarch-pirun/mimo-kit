# PRD: 编程终端智能体系统

## 可行性判断
结论: 值得做
理由: 现有 AI 编程工具能力碎片化，缺少统一调度+监督机制。"AI 监督 AI"方向差异化明显，且用户自身痛点明确、范围可控。

---

## 需求清单

| ID | 需求 | 验收标准 | 严重度 |
|----|------|---------|--------|
| R1 | **工具接入层** — 至少接入 2 个智能体（如 Claude Code + Codex），并为扩展留接口 | 两个工具都能被正常调用并返回结果 | P0 |
| R2 | **任务理解与拆解** — 用户一句话指令 → 系统拆解为可执行的子任务链 | "帮我把项目 A 迁移到项目 B" 能正确拆出 3+ 子任务 | P0 |
| R3 | **调度决策层** — 根据子任务类型自动选择最合适的智能体执行 | 编码任务走 Codex，审查任务走 Claude Code 或反之 | P0 |
| R4 | **监督与审查层** — 一个智能体监督另一个的执行结果，做代码审查 | Codex 写完代码 → Claude Code 审查并给出修复意见；反之亦然 | P0 |
| R5 | **争议仲裁** — 监督者与被监督者分歧时的解决机制 | 分歧时由第三智能体投票或交由用户仲裁 | P1 |
| R6 | **执行回显** — 实时显示每个智能体的状态、进度、产出 | 用户在终端看到"Codex: 生成中 / Claude Code: 审查中..." | P1 |
| R7 | **记忆与复用** — 成功任务路径缓存，下次类似任务自动调用经验 | 同一类任务第二次调用时执行时间缩短 30% | P2 |
| R8 | **监督结果沉淀** — 审查记录入库，提炼为可复用规则 | 10 次审查后自动检出 1 条可复用的监督模式 | P2 |

---

## 不做的事

- ❌ 不做图形界面（纯终端）
- ❌ 不做智能体自身的训练或调优
- ❌ 不做与现有 CI/CD 工具的集成（Phase 2 考虑）

---

## 核心风险

| 风险 | 严重度 | 缓解策略 |
|------|--------|---------|
| 目标偏移 — 监督者改错、放错 | P0 | R4 必须有争议仲裁兜底，早期人工参与仲裁 |
| 上下文损耗 — 1+1 < 2（AI 间传递信息丢失/扭曲原始意图） | P0 | 最小闭环先验证：仅 2 个智能体、单一任务链 |
| 元认知消耗 — 调度/监督开销超过收益 | P1 | 基准测试：对比单人 AI vs 双人 AI 的完成时间/质量 |

---

## 认知对话摘要

- **发散阶段覆盖**: 黄金圈法则、第一性原理
- **收敛阶段覆盖**: MECE、逻辑树
- **验证阶段覆盖**: 逆向思维、批判性思维
- **关键转折点**: 用户明确"监督与审查"是核心差异（非技术调度本身，而是 AI 监督 AI）
- **核心风险锚定**: "1+1 < 2" 为目标偏移+上下文损耗+元认知消耗的复合问题
