# /brief — 需求理解与锚定

## 它做什么

原版交互全部保留。区别：
- 原版：需求确认完直接写 PRD.md
- 新版：需求确认完写 `confirmed-requirements.md`（原子化需求清单）
- 加料：第 4 步检查粒度时从开放式提问改为**给选项**

---

## 流程

### 第一步：听你说（不改）

你随意说。我记下来。不问框架问题。

### 第二步：我复述（不改）

整理成需求点，一条一条复述，等你确认。

### 第三步：你纠正（不改）

你说不对我就改，直到你说"对，就是这个意思"。

### 第四步：选项式检查粒度（改）

当你确认了一条需求后，如果有模糊点，**不给开放式问题，给选项**：

```
原版:
"你刚确认了用户注册——你说的注册包括哪些？"

新版:
"用户注册的方式？
  A. 邮箱 + 密码注册
  B. 手机号 + 验证码
  C. 第三方登录（微信/GitHub）
  D. 以上全部
  E. 其他"
```

选项模板（按需求类型匹配）：

| 需求类型 | 选项方向 | 例子 |
|---------|---------|------|
| 用户相关 | 注册/登录/权限方式 | A.邮箱 B.手机 C.第三方 D.其他 |
| 触发方式 | 自动/手动/条件 | A.自动触发 B.手动触发 C.条件触发 |
| 数据相关 | 存储位置/格式 | A.本地 SQLite B.云端 C.不存 |
| 交互方式 | UI/CLI/API | A.网页 B.命令行 C.API 接口 |
| 集成方式 | 内部/外部 | A.自建 B.调第三方 C.两者 |

**规则：**
- 一次只问一个点，给 2-4 个选项
- 选项包含"其他"兜底
- 用户如果选"其他"并说了自己的方案，继续下一条，不追问
- 用户如果说不确定，标记"待定"，继续下一条

**为什么改成给选项：** 开放式问题要靠用户自己想。给选项是"你认不认"，省力且不遗漏。

### 第五步：收口（不改）

所有需求确认完后问："你还有什么想说但刚才没说的？"
说了有 → 回到第一步。说了"没有了" → 走下一步。

### 第六步：产出 confirmed-requirements.md（改）

写 `spec/confirmed-requirements.md`。

格式：每行一条原子需求，自然语言，带编号。

```
R1: 用户能用邮箱和密码注册新账号
R2: 用户能用邮箱和密码登录
R3: 提交 PR 后自动触发代码审查
R4: 审查结果以评论形式贴回 PR
```

**原子化规则：**
- 一条需求只包含一个功能点
- 用户说"注册和登录" → 拆成 R1 注册 + R2 登录
- 用户说"审查代码并贴结果" → 拆成 R3 触发审查 + R4 贴结果

---

## 约束（不改）

- 不预设框架
- 不跳过确认
- 不产出 PRD（只产出 confirmed-requirements.md）
- 追问只追一个点，给选项

## 输出

```
✅ 需求锚定完成 → spec/confirmed-requirements.md

共确认 N 条需求:
  R1: ...
  R2: ...
  R3: ...

下一步: /detail 逐条展开颗粒度
```
